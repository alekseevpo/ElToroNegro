# Руководство по ведению журнала разработки

## Принципы ведения журнала

### Когда обновлять CHANGELOG.md

**ОБЯЗАТЕЛЬНО обновлять при:**
- ✅ Добавлении новой функции/файла
- ✅ Изменении существующей функциональности
- ✅ Исправлении багов
- ✅ Изменении документации (значимых)
- ✅ Добавлении/удалении зависимостей
- ✅ Изменении конфигурации проекта
- ✅ Изменении структуры проекта

**Можно пропустить:**
- ❌ Мелкие исправления опечаток
- ❌ Рефакторинг без изменения функциональности (можно упомянуть кратко)

### Формат записей

```markdown
## [Версия] - YYYY-MM-DD

### Добавлено
- Описание новой функции

### Изменено
- Что было изменено и почему

### Исправлено
- Описание исправленного бага

### Безопасность
- Изменения, связанные с безопасностью
```

### Структура записей

1. **Заголовок версии** - используйте Semantic Versioning (MAJOR.MINOR.PATCH)
2. **Дата** - дата изменения в формате YYYY-MM-DD
3. **Категории изменений:**
   - **Добавлено** - новые функции
   - **Изменено** - изменения существующих функций
   - **Устарело** - функции, помеченные как deprecated
   - **Удалено** - удаленные функции
   - **Исправлено** - исправления багов
   - **Безопасность** - изменения безопасности

### Пример хорошей записи

```markdown
## [0.2.0] - 2024-01-15

### Добавлено
- Emergency Pause механизм в LotteryV2.sol
- Функция возврата средств `refundMyTickets()`
- Настраиваемая комиссия для owner (максимум 10%)
- Таймауты для лотереи и выбора победителя

### Изменено
- README.md обновлен с информацией о критических предупреждениях
- Функция `getLotteryStatus()` теперь возвращает информацию о таймаутах

### Технические детали
- Интегрирован OpenZeppelin Pausable и Ownable
- Добавлены новые события: FundsRefunded, EmergencyRefund
```

### Контроль журнала

**При каждой сессии работы:**
1. Проверьте раздел [Unreleased]
2. Перенесите выполненные задачи в соответствующую версию
3. Обновите дату последнего обновления
4. Убедитесь, что все значимые изменения задокументированы

**Перед коммитом:**
1. Проверьте, что все изменения отражены в CHANGELOG.md
2. Убедитесь, что формат корректен
3. Обновите версию, если требуется

### Рекомендации

- **Будьте конкретны** - описывайте что было сделано, а не просто "улучшения"
- **Группируйте** - объединяйте связанные изменения
- **Используйте ссылки** - ссылайтесь на issues/PR, если они есть
- **Будьте последовательны** - используйте один стиль записи
